# SPEC-DOCS-001: 수용 기준

> 커스터마이징 가이드 확장을 위한 수용 기준 (Given-When-Then 형식)
>
> **범위**: 홈페이지 커스터마이징, 커스텀 페이지 추가, 플러그인 유형 차이, 업데이트 보존, 안전 지침

---

## 1. 개요 (Overview)

### 1.1 수용 기준 개요

이 문서는 SPEC-DOCS-001의 요구사항을 충족하는지 검증하기 위한 Given-When-Then 형식의 수용 기준을 정의합니다.

### 1.2 확장된 범위

이 SPEC은 다음과 같은 확장 범위에 대한 수용 기준을 포함합니다:

1. **커스텀 페이지 추가**: new-route 플러그인 생성 및 배포
2. **플러그인 유형 선택**: new-route vs override 비교 및 선택
3. **업데이트 보존 검증**: core:pull 후 커스터마이징 유지 확인
4. **안전 지침 준수**: 안전/위험 작업 구분 및 롤백 가능성

### 1.2 테스트 전략

- **수동 테스트**: 문서 내용 검토, 사용자 테스트
- **자동화 테스트**: LLM을 사용한 문서 유효성 검증
- **사용자 테스트**: 실제 로컬 클라이언트를 통한 사용성 테스트

---

## 2. 핵심 요구사항 수용 기준

### AC-DOCS-001: 전용 홈페이지 커스터마이징 가이드 생성

#### Scenario 1: 가이드 문서가 존재하고 접근 가능

**Given** 사용자가 `/docs/` 디렉토리에 접근할 때
**And** `HOMEPAGE_CUSTOMIZATION_GUIDE.md` 파일이 존재할 때
**When** 사용자가 파일을 열람할 때
**Then** 파일이 정상적으로 로드되어야 한다
**And** 파일이 Markdown 형식이어야 한다
**And** 파일이 이중 언어(한국어/English)로 작성되어야 한다

#### Scenario 2: 가이드 문서의 주요 섹션이 포함되어 있음

**Given** `HOMEPAGE_CUSTOMIZATION_GUIDE.md` 파일이 열려 있을 때
**When** 사용자가 문서를 스크롤할 때
**Then** 다음 6개 주요 섹션이 모두 존재해야 한다:
  - 빠른 시작 (Quick Start)
  - 섹션 레퍼런스 (Section Reference)
  - 자연어 프롬프트 예시 (Natural Language Prompts)
  - 테스트 및 검증 (Testing & Validation)
  - 문제 해결 (Troubleshooting)
  - 고급 커스터마이징 (Advanced)

#### Scenario 3: 가이드가 비기술적 사용자를 대상으로 함

**Given** 비기술적 배경을 가진 사용자가 가이드를 읽을 때
**When** 사용자가 빠른 시작 섹션을 읽을 때
**Then** 기술적 전문 지식 없이도 이해할 수 있어야 한다
**And** 용어에 대한 설명이 제공되어야 한다
**And** 단계별 지침이 명확해야 한다

### AC-DOCS-002: SectionRenderer API 문서화

#### Scenario 1: 모든 섹션 타입이 문서화됨

**Given** 사용자가 섹션 레퍼런스 섹션을 열람할 때
**When** 사용자가 섹션 목록을 확인할 때
**Then** SectionRenderer가 지원하는 모든 섹션 타입이 문서화되어야 한다
**And** 최소 30개 이상의 섹션 타입이 포함되어야 한다

**검증 방법**:
```bash
# SectionRenderer.astro의 섹션 타입 수 확인
grep -o "Section," src/components/common/SectionRenderer.astro | wc -l

# 가이드 문서의 섹션 타입 수 확인
grep -c "^###" /docs/HOMEPAGE_CUSTOMIZATION_GUIDE.md
```

#### Scenario 2: 각 섹션의 Props가 문서화됨

**Given** 사용자가 특정 섹션 타입을 조회할 때
**When** 사용자가 섹션의 상세 정보를 확인할 때
**Then** 다음 항목이 포함되어야 한다:
  - 타입 ID (예: `MainHero`, `BridgeSection`)
  - 용도 설명
  - 필수 Props 테이블
  - 선택적 Props 테이블
  - 사용 예시 코드
  - 주의사항

#### Scenario 3: Props 테이블이 표준 형식을 따름

**Given** 사용자가 섹션의 Props를 확인할 때
**When** 사용자가 Props 테이블을 읽을 때
**Then** 테이블은 다음 열을 포함해야 한다:
  - Prop (속성명)
  - 타입 (Type)
  - 설명 (Description)
  - 기본값 (Default)

**예시 테이블**:
| Prop | 타입 | 설명 | 기본값 |
|------|------|------|--------|
| images | Array<string> | 히어로 이미지 URL 목록 | [] |
| mainHeading | string | 메인 제목 텍스트 | "" |

#### Scenario 4: 각 섹션에 사용 예시가 포함됨

**Given** 사용자가 섹션 문서를 읽을 때
**When** 사용자가 실제 적용 방법을 알고 싶을 때
**Then** 각 섹션에 최소 1개 이상의 사용 예시가 포함되어야 한다
**And** 예시는 실행 가능한 코드여야 한다
**And** 예시에는 주석이 포함되어야 한다

### AC-DOCS-007: 플러그인 유형별 차이점 문서화

#### Scenario 1: 플러그인 유형 비교 테이블이 제공됨

**Given** 사용자가 플러그인 유형을 선택하려 할 때
**When** 사용자가 비교 테이블을 확인할 때
**Then** 다음 항목이 비교되어야 한다:
  - 목적 (새 페이지 vs 기존 페이지 교체)
  - URL 경로 (/ext/{plugin-id} vs 원본 경로)
  - Priority 값 (5 vs 10)
  - 충돌 처리 방식
  - 사용 사례

#### Scenario 2: 선택 결정 트리가 제공됨

**Given** 사용자가 어떤 플러그인 유형을 선택할지 결정해야 할 때
**When** 사용자가 결정 트리를 따를 때
**Then** 다음 질문으로 유형이 결정되어야 한다:
  1. 새로운 페이지를 추가하나요? → YES: new-route
  2. 기존 페이지를 완전히 교체하나요? → YES: override
  3. 기존 페이지의 일부만 수정하나요? → override에서 섹션만 변경

#### Scenario 3: 각 유형별 충돌 처리가 설명됨

**Given** 사용자가 여러 플러그인을 사용할 때
**When** 플러그인 간 충돌이 발생할 때
**Then** 충돌 처리 방식이 설명되어야 한다:
  - new-route: 독립 경로로 충돌 없음
  - override: priority가 높은 플러그인이 우선
  - 동일 priority: 경고 메시지 표시

### AC-DOCS-008: 업데이트 보존 프로세스 문서화

#### Scenario 1: 보호된 위치가 문서화됨

**Given** 사용자가 `core:pull` 업데이트를 실행할 때
**When** 사용자가 어떤 파일이 보존되는지 확인할 때
**Then** 다음 위치가 문서화되어야 한다:
  - `src/plugins/local/` - 로컬 플러그인
  - `src/survey-tools/local/` - 로컬 설문 도구
  - `src/lib/local/` - 로컬 유틸리티
  - `public/local/` - 로컬 에셋

#### Scenario 2: 4레벨 필터 시스템이 설명됨

**Given** 사용자가 업데이트 필터링을 이해하려 할 때
**When** 사용자가 필터 시스템을 확인할 때
**Then** 다음 4레벨이 설명되어야 한다:
  1. **LOCAL_PREFIXES**: `local/` 접두사 파일은 항상 보존
  2. **PROTECTED_EXACT**: 정확히 일치하는 파일은 백업 후 복원
  3. **LOCAL_PATH_PARTS**: 경로에 `local`이 포함된 파일 보존
  4. **CORE_REPLACEMENTS**: 코어 파일만 교체

#### Scenario 3: 업데이트 후 확인 체크리스트가 제공됨

**Given** 사용자가 업데이트를 완료했을 때
**When** 사용자가 커스터마이징이 유지되었는지 확인할 때
**Then** 확인 체크리스트가 제공되어야 한다:
