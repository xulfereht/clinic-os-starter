# /setup-clinic-info — Clinic Basic Information Setup

> **Role**: Clinic Setup Manager
> **Cognitive mode**: Extract-first, ask-second. Use every available data source to pre-fill, then confirm with the user.

Sets up clinic name, contact, address, business hours, and other basic information.
This is a core step in Onboarding Tier 1, callable from `/onboarding` or standalone.

## When to Use

- First step of any clinic setup — basic info entry
- After `/extract-content` provides Place data — auto-map fields
- When clinic info changes — address, phone, hours update
- Triggered by onboarding Tier 1

## Data Sources (priority order)

1. `.agent/clinic-profile.json` — Place/blog profile generated by `/extract-content` (highest priority)
2. `GET /api/clinic-info.ts` — Current DB settings
3. `GET /api/settings.ts` — Full site settings
4. User direct input — Only for items missing from above sources

## Procedure

### Step 1 — Collect data sources (automatic)

```bash
# Priority 1: clinic-profile.json (extract-content result)
cat .agent/clinic-profile.json 2>/dev/null

# Priority 2: Current DB settings
npx wrangler d1 execute DB --local --command \
  "SELECT category, key, value FROM site_settings WHERE category IN ('general', 'contact', 'info') ORDER BY category, key;"
```

### Step 1.5 — Profile completeness gate (모든 모드 공통)

clinic-profile.json 수집 후 충족도를 검증합니다. 이 게이트는 데모/위임/직접설치 모두 동일하게 작동합니다.

```bash
# 필수 필드 검증
node -e "
  const p = JSON.parse(require('fs').readFileSync('.agent/clinic-profile.json'));
  const checks = [
    ['contact.phone', p.contact?.phone],
    ['contact.address', p.contact?.address],
    ['clinic.representative', p.clinic?.representative],
    ['hours.weekdays', p.hours?.weekdays],
  ];
  const missing = checks.filter(([k,v]) => !v).map(([k]) => k);
  if (missing.length) {
    console.log('⚠️  clinic-profile 불완전:', missing.join(', '));
    process.exit(1);
  }
  console.log('✅ clinic-profile 충족');
"
```

**불완전할 경우 대응:**
- `source.type === "demo_placeholder"` 또는 `source.url === null` → `/extract-content` 실행 권장: "네이버 플레이스 URL을 알려주시면 자동으로 정보를 채울 수 있습니다."
- 이미 extract 했는데 빠진 필드 → 사용자에게 해당 필드만 직접 입력 요청
- 사용자가 skip 요청 → 허용하되 경고: "더미 데이터가 사이트에 노출될 수 있습니다."

**site_settings 더미 감지:**
```sql
SELECT key, value FROM site_settings
WHERE value IN ('샘플한의원', '02-1234-5678', '서울특별시 강남구 테헤란로 123');
```
더미 값이 존재하면 반드시 덮어쓰기 실행.

### Step 2 — Auto-mapping

If clinic-profile.json exists, auto-map the following fields:

| clinic-profile Field | → DB Location | Notes |
|---------------------|-----------|------|
| `clinic.name_ko` | `general.site_name` | 한의원 이름 |
| `clinic.representative` | `general.representative` | 대표원장 |
| `clinic.phone` | `contact.phone` | 전화번호 |
| `clinic.address` | `contact.address` | 주소 |
| `clinic.business_hours` | `info.business_hours` | 영업시간 (JSON) |
| `clinic.categories` | (reference only) | 진료과목 힌트 |
| `clinic.naver_place_id` | `contact.naver_place` | 플레이스 링크 |
| `blog.author_name` | `general.representative` | 블로그 작성자 (fallback) |

**Mapping principles:**
- If value exists in clinic-profile → auto-fill
- If DB already has a different value → show conflict, ask user to choose
- If neither has value → ask user

### Step 4 — Apply to DB (batch)

Apply all confirmed information at once:

```bash
npx wrangler d1 execute DB --local --command \
  "INSERT OR REPLACE INTO site_settings (category, key, value) VALUES
   ('general', 'site_name', '{name}'),
   ('general', 'representative', '{representative}'),
   ('contact', 'phone', '{phone}'),
   ('contact', 'address', '{address}'),
   ('info', 'business_hours', '{hours_json}');"
```

**Caution**: When writing to production DB, always get user confirmation before running with `--remote`.

### Step 4.5 — clinics 테이블 동기화

`getClinicSettings()`는 `clinics` 테이블을 먼저 읽고 `site_settings`로 보완합니다.
두 테이블의 데이터가 다르면 **clinics 값이 우선 표시**되어 의도치 않은 결과가 발생합니다.

**반드시 site_settings와 clinics를 동시에 업데이트합니다.**

```sql
-- clinics 테이블 동기화 (site_settings와 동일한 값으로)
UPDATE clinics SET
  name = '{name}',
  address = '{address}',
  full_address = '{address}',
  phone = '{phone}',
  representative_name = '{representative}',
  updated_at = unixepoch()
WHERE id = (SELECT id FROM clinics LIMIT 1);
```

**동기화 검증:**
```sql
-- 불일치 감지 쿼리
SELECT
  c.address AS clinics_address,
  s.value AS settings_address,
  CASE WHEN c.address = s.value THEN '✅' ELSE '❌ MISMATCH' END AS status
FROM clinics c
CROSS JOIN (SELECT value FROM site_settings WHERE key = 'address') s;
```

**logo_url 동기화:**
- clinics.logo_url이 `'none'` → 텍스트 로고 사용 (의도적)
- clinics.logo_url이 실제 경로 → 해당 파일이 `public/` 에 존재하는지 확인
- 파일 미존재 시 → `logo_url = 'none'`으로 리셋하고 사용자에게 알림

```sql
-- logo_url 파일 존재 확인 후, 없으면 텍스트 로고로 리셋
-- (agent가 파일 존재 여부를 ls로 확인 후 실행)
UPDATE clinics SET logo_url = 'none'
WHERE logo_url IS NOT NULL
  AND logo_url != 'none'
  AND logo_url NOT IN (/* verified existing paths */);
```

3. If direct input → ask item by item (legacy approach)
